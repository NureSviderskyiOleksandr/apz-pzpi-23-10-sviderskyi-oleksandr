3 ОПИС ВИКОНАНОЇ РОБОТИ
   3.1 Загальні відомості про систему
• Назва системи: Figma.
• Розробник: Figma, Inc. (зараз у складі Adobe).
• Призначення системи: Хмарний інструмент для дизайну інтерфейсів та прототипування з можливістю спільної роботи в реальному часі. Використовується UI/UX дизайнерами, продуктовими менеджерами та розробниками для створення макетів, іконок та графічних елементів прямо у браузері або десктопному додатку.
   3.2 Архітектурний стиль системи
Figma використовує гібридний архітектурний стиль:
• Клієнт-серверна архітектура: основна частина обчислень (рендеринг графіки) винесена на сторону клієнта.
• Мікросервісна архітектура (Backend): серверна частина складається з набору незалежних сервісів (авторизація, управління файлами, платіжні системи).
• Подієво-орієнтована (Event-driven): використовується для синхронізації дій багатьох користувачів у реальному часі через WebSocket.
     Обґрунтування: Вибір зумовлений необхідністю забезпечити швидкість відгуку "нативного" додатка у веб-браузері. Використання WebAssembly дозволяє перенести бізнес-логіку рендерингу на клієнта, мінімізуючи затримки мережі.
     3.3 Основні компоненти системи
• Інтерфейс користувача (Frontend): написаний на React та TypeScript для побудови панелей управління. Проте саме графічне полотно (Canvas) використовує C++, скомпільований у WebAssembly (Wasm), що забезпечує продуктивність на рівні десктопних програм.
• Серверна частина (Backend): написана переважно на Go. Використовується для обробки запитів, керування правами доступу та координації сесій.
• Рівень бізнес-логіки: розподілений. Частина логіки (редагування об'єктів) працює на клієнті через Wasm-рушій, а частина (координація правок) — на спеціальних Multiplayer Servers (написаних на Rust для максимальної швидкості).
• Рівень зберігання даних: PostgreSQL для метаданих та користувачів, S3 для зберігання великих бінарних файлів (зображень) та Redis для кешування сесій.
• Допоміжні сервіси: сервіси експорту (перетворення векторів у PNG/SVG), сервіси генерації прев'ю.
3.4 Взаємодія компонентів
     Figma використовує протокол WebSocket для постійного з'єднання між клієнтом та сервером синхронізації. 
     Механізм синхронізації, замість класичного Operational Transformation (OT), використовується власна реалізація, близька до CRDT (Conflict-free Replicated Data Types). Це дозволяє декільком користувачам редагувати один і той самий векторний об'єкт без конфліктів.
     Протоколи: HTTPS для завантаження початкових даних та WebSocket для передачі диференціальних оновлень (binary diffs).

Рисунок 3.1 — UML-діаграма компонентів сервісу Figma
Опис діаграми компонентів:
1. Client Tier (Frontend): складається з двох частин. UI Layer (React) відповідає за меню, налаштування та панелі інструментів. Rendering Engine (WebAssembly/C++) виконує складні математичні обчислення та відмальовує векторну графіку через WebGL.
2. Multiplayer Tier (Real-time): високопродуктивний сервер на Rust, який тримає активні WebSocket-з'єднання. Його завдання — миттєво ретранслювати зміни (operations) від одного користувача іншим, зберігаючи їх у тимчасовому кеші.
3. Application Tier (Backend): написаний на Go, обробляє стандартні запити: аутентифікація, перегляд списку проектів, керування командами.
4. Data Tier (Persistence): PostgreSQL зберігає структуру файлів та права доступу, Amazon S3 — важкі бінарні дані (картинки, шрифти), а Redis — активні сесії.
3.5  Масштабованість та надійність
     Figma використовує Nginx та AWS ELB для розподілу трафіку між мікросервісами. Multiplayer-сервери ізольовані за файлами, якщо один сервер впаде, це вплине лише на користувачів конкретних файлів, а не на весь сервіс.
     Агресивне кешування у Figma відбувається на рівні браузера (IndexedDB), це дозволяє працювати з великими файлами навіть при нестабільному інтернеті.
     Слід зазначити, що система повністю розгорнута в AWS, що дозволяє динамічно масштабувати кількість серверів залежно від кількості активних сесій.
     3.6 Приклади програмної реалізації
     Для демонстрації архітектурного принципу синхронізації, розглянемо спрощений концепт CRDT на TypeScript, який Figma використовує для уникнення конфліктів при зміні координат об'єкта (див. Додаток В).
     Наданий фрагмент коду демонструє принцип детермінованого злиття даних без участі центрального арбітра. У Figma кожна властивість об'єкта (координати, колір, назва) розглядається як окремий регістр із часовою міткою.
1. Структура даних VectorNode
Замість того, щоб зберігати координати як прості числа (x: 100), система зберігає їх як об'єкти з метаданими:
• value: безпосереднє значення координати.
• timestamp: логічний час (Lamport Clock). Вона вказує не на реальний час на годиннику, а на порядковий номер операції у розподіленій системі.
2. Алгоритм Last Write Wins (LWW)
Функція mergeChanges реалізує правило «Останній запис перемагає». Це найпростіший тип CRDT, який гарантує збіжність (Convergence):
• При отриманні змін від іншого користувача (remote), клієнт порівнює мітки часу для кожного поля окремо.
• Якщо remote.timestamp > local.timestamp, локальне значення замінюється.
• Якщо два користувачі змінили різні поля (наприклад, один — x, інший — y), система об'єднає обидві зміни. Це дозволяє уникнути конфліктів, які зазвичай виникають у класичних системах при повному перезаписі об'єкта.

1 // Спрощений приклад структури об'єкта з LWW (Last Write Wins) елементом
2 interface VectorNode {
3     id: string;
4     properties: {
5         x: { value: number, timestamp: number };
6         y: { value: number, timestamp: number };
7     };
8 }
9 
10 // Функція злиття змін від різних користувачів
11 function mergeChanges(local: VectorNode, remote: VectorNode): VectorNode {
12     return {
13         id: local.id,
14         properties: {
15             x: remote.properties.x.timestamp > local.properties.x.timestamp 
16                ? remote.properties.x : local.properties.x,
17             y: remote.properties.y.timestamp > local.properties.y.timestamp 
18                ? remote.properties.y : local.properties.y
19         }
20     };
21 }
